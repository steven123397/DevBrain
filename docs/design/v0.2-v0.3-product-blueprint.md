# DevBrain v0.2-v0.3 产品蓝图

文档状态：draft
最后更新：2026-04-19
负责人：Hermes

关联计划：`docs/plan/v0.2-ai-assist-layer.md`
关联状态：`docs/status/project-status.md`

## 1. 背景

`docs/design/product-requirements.md` 已给出 DevBrain 的长期路线：
- v0.1：Knowledge Cards MVP
- v0.2：AI Assist Layer
- v0.3：Knowledge Graph Layer
- v0.4+：Context & Repo Layer

当前 v0.1 已基本跑通「收集 -> 整理 -> 搜索 -> 复用」主闭环，但如果不先把后续阶段的边界写清楚，后续很容易出现以下问题：
- 把 AI 辅助、embedding、图谱、决策对象混成一个模糊升级包。
- 让尚未稳定的搜索 / 召回底层承受过早的上层设计压力。
- 让未来实现讨论失去“什么该先做、什么不该越级做”的清晰边界。

因此，需要一份单独的产品蓝图，用来定义 v0.2 与 v0.3 的阶段目标、非目标、承接关系与推进前提。

## 2. 目标

- 明确 v0.2 与 v0.3 分别解决什么问题，而不是只按技术名词切版本。
- 定义 AI Assist Layer 与 Knowledge Graph Layer 的阶段边界。
- 让 v0.2 / v0.3 的讨论建立在 v0.1 已验证过的产品价值之上。
- 与 `docs/design/future-data-model.md` 保持一致：坚持 `note-first, graph-later` 的演进原则。
- 为未来计划文档和实施讨论提供稳定的产品口径。

## 3. 非目标

- 不是当前迭代的实现计划，不拆解具体任务和文件改动。
- 不把 AI 设定为主流程替代者，也不把 agent-first 交互定义为默认方向。
- 不在本阶段展开多用户、云同步、协作、权限模型等产品路线。
- 不把 v0.3 的图谱可视化当作优先于知识质量治理的目标。
- 不提前把 v0.4 的 Context / Repo Layer 拉进当前阶段范围。

## 4. 方案边界

本蓝图只覆盖：
- v0.2：AI Assist Layer
- v0.3：Knowledge Graph Layer

相关但不展开：
- v0.1 的验证设计：由 `docs/design/v0.1-validation-lab.md` 负责。
- 搜索 / 召回升级门槛：由 `docs/design/search-and-recall-evolution.md` 负责。
- 更长期的数据实体与关系草案：由 `docs/design/future-data-model.md` 负责。
- 更后期的合成成长实验：由 `docs/design/synthetic-growth-lab.md` 负责。

## 5. 关键决策

- 决策 A：v0.2 定义为 AI Assist Layer，而不是 AI-first 产品重构。
  - AI 的角色是辅助结构化提取、增强召回、压缩历史知识，而不是替代原有笔记主流程。

- 决策 B：v0.3 定义为 Knowledge Graph Layer，而不是“先做图谱可视化”。
  - 核心是把知识对象和关系显式化，视图只是结果的一部分，不是全部。

- 决策 C：坚持 `note-first, graph-later`。
  - v0.1 的 Note 主模型仍是所有上层能力的基础；v0.2 和 v0.3 都不应推翻它。

- 决策 D：v0.2 / v0.3 的推进必须建立在 v0.1 验证成立之上。
  - 如果搜索、Digested 质量门槛、相关推荐价值尚不稳定，上层能力只会放大底层噪声。

- 决策 E：Context / Repo Layer 继续后置到 v0.4+。
  - 在知识对象和关系尚未稳定前，过早引入项目 / 仓库 / 文件绑定会导致系统边界失焦。

## 6. v0.2：AI Assist Layer

### 6.1 阶段目标

v0.2 的核心目标不是“让系统更像聊天机器人”，而是让现有知识卡片系统更容易被整理、找回和复用。

它主要解决 3 类问题：
- 用户有知识，但结构化整理成本仍偏高。
- 用户有搜索结果，但难以快速理解哪条更值得复用。
- 用户面对模糊记忆场景时，关键词召回仍有天然上限。

### 6.2 核心能力

v0.2 可优先考虑以下能力：
- **AI 自动摘要**：为长 `raw_input` 或已整理条目生成更紧凑的摘要候选。
- **AI 标签建议**：辅助建议标签与技术栈，而不是替用户强制分类。
- **结构字段候选提取**：从 `raw_input` 中提取 `problem / solution / why` 候选内容。
- **embedding 相似召回**：在关键词不完全重叠时补充近义候选。
- **搜索 / 详情压缩提示**：对召回结果做轻量压缩，帮助用户更快判断是否值得打开。

### 6.3 非目标

v0.2 明确不做：
- 让 AI 成为系统可用性的前提。
- 以聊天入口替代列表、详情、编辑与结构化整理。
- 自动替用户完成全部整理决策而不给出可审阅候选。
- 把 AI 结果直接视为可信知识而不经过用户确认。

### 6.4 推进前提

v0.2 需要以下前提至少基本成立：
- v0.1 的搜索和相关推荐已被验证为有基础价值。
- 当前 Digested 质量门槛基本可用。
- 搜索 / 召回升级路径已经定义清楚，知道 AI 介入是在增强什么，而不是补救什么。

## 7. v0.3：Knowledge Graph Layer

### 7.1 阶段目标

v0.3 的核心目标是：把当前主要依赖 Note 文本和轻量字段的知识系统，进一步发展为具备显式对象与关系的知识网络。

它要解决的问题是：
- 当前系统能存和搜知识，但对“知识之间有什么关系”表达能力有限。
- 用户能复用解决方案，但尚难系统性复用“决策路径”“适用边界”“相关概念群”。
- 相关推荐仍主要停留在“相关”，缺少更明确的结构语义。

### 7.2 核心能力

v0.3 可优先考虑以下方向：
- **显式关系边**：从“相关”逐步过渡到 `solves`、`depends_on`、`similar_to` 等更明确关系。
- **知识对象显式化**：在 Note 之外，逐步显式化 `Concept / Problem / Solution / Procedure / Decision / Context` 等对象。
- **图谱浏览**：支持围绕某个问题、概念或方案向外探索知识邻域。
- **主题聚类与连接发现**：帮助用户看到知识团块与连接缺口。
- **决策路径沉淀**：逐步让“为什么这样选、没选什么、适用边界是什么”成为可复用对象。

### 7.3 非目标

v0.3 明确不做：
- 只做炫目的图可视化而缺少稳定语义基础。
- 把图结构当成绕过搜索和输入治理问题的捷径。
- 在实体和关系尚未稳定前，过早扩展到大规模 repo / file 绑定。
- 用复杂图模型替代当前仍然有效的 Note 主模型。

### 7.4 与未来数据模型的关系

v0.3 的实体与关系方向，应与 `docs/design/future-data-model.md` 保持一致：
- 先从 Note 中抽取稳定对象。
- 再逐步沉淀对象之间的显式关系。
- 最终为更后续的 Context / Repo Layer 和 Capability Layer 留出空间。

## 8. 阶段承接关系

### 8.1 从 v0.1 到 v0.2

前提不是“已经有 AI 能接”，而是：
- 已确认当前知识卡片闭环具备产品价值。
- 已知道搜索 / 相关推荐的主要瓶颈是什么。
- 已能区分哪些问题应由结构化治理解决，哪些问题值得引入 AI 增强。

### 8.2 从 v0.2 到 v0.3

前提不是“已经接入模型”，而是：
- AI 辅助已帮助系统更稳定地产生结构化知识候选。
- 搜索与召回底层已足够稳定，不再过度依赖手工补救。
- 已观察到用户确实需要更显式的关系、对象和决策路径，而不只是更多 note。

## 9. 风险与权衡

- 风险 1：过早把 AI 做成主流程，削弱本地优先和结构化主线。
  - 权衡：v0.2 只做辅助层，不取代主流程。

- 风险 2：过早推进图谱，让系统在知识质量尚不稳定时变得更复杂。
  - 权衡：v0.3 先建立对象和关系语义，再谈更丰富视图。

- 风险 3：把版本路线按技术名词划分，而不是按用户价值划分。
  - 权衡：每个阶段都要回答“它让用户更容易完成哪一步”。

- 风险 4：让未来设计反向干扰当前主线。
  - 权衡：这份蓝图用于定义边界，不自动进入当前实现排期。

## 10. 验证方式

当前设计文档本身不要求实现级验证；待用户拍板进入后续阶段时，建议按以下方式验证：

- 验证项 1：v0.2 候选能力是否真的降低整理成本或提升模糊召回能力。
- 验证项 2：AI 辅助结果是否可审阅、可拒绝、可修正，而不是黑盒替代。
- 验证项 3：v0.3 引入的对象和关系是否帮助用户理解知识网络，而不是增加维护负担。
- 验证项 4：各阶段推进前，先确认上一阶段的验证结论已成立。
