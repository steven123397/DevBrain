## 1. 背景

`docs/design/product-requirements.md` 与 `docs/design/future-data-model.md` 已经明确了 DevBrain 的长期方向：
- 从结构化知识卡片继续生长为知识网络
- 逐步支持决策路径、上下文召回与能力地图
- 最终验证系统是否真的会随着时间积累而变得更有用，而不只是堆积更多笔记

问题在于，这类能力的真实验证天然依赖长时间跨度的数据积累。
如果只靠真实个人使用，需要数月甚至更长时间，才能观察到：
- 知识是否会自然形成网络
- 决策路径是否会持续沉淀
- 搜索与相关推荐是否会随着数据增长而退化或成熟
- 系统是否能在不同记录风格下保持可消化、可生长，而不是逐渐变脏、变乱、变难用

因此需要一个未来实验方案：
在云上部署一个或多个 DevBrain 与 Hermes worker，让 Hermes 持续模拟真实程序员的工作、记录、整理、检索、复用与决策行为，以较短时间压缩观察长周期知识成长过程。

这份文档只负责记录该未来实验方向，帮助后续在知识图谱 / 决策对象 / 情境召回等能力成熟后，再讨论是否立项。
它不是 MVP，也不是当前阶段的执行依据。

## 2. 目标

- 为 DevBrain 建立一个“合成成长实验场”的长期设计基线。
- 验证 DevBrain 在长时间、高频、多风格输入下，是否能长成可消化、可生长的知识网络。
- 验证系统是否能沉淀出有解释性的决策路径、上下文关联与召回价值。
- 为未来的搜索、相关推荐、知识图谱、决策对象、能力地图等机制提供受控实验环境。
- 在不等待真实个人使用数月的前提下，快速暴露产品在结构、质量与维护性上的缺陷。

## 3. 非目标

- 不是 v0.1 / 当前 MVP 的交付内容。
- 不是当前阶段的开发优先级，不应挤占知识图谱之前的主线资源。
- 不是为了伪造漂亮指标或制造“看起来很多数据”的假繁荣。
- 不是一开始就做真实模型微调、RL 或复杂训练平台。
- 不是当前生产部署方案，也不是正式多用户云服务路线图。

## 4. 方案边界

本方案只讨论未来实验基础设施，不规定当前代码库必须实现的模块。

覆盖范围：
- 云上运行的 DevBrain 实验实例
- 模拟真实程序员的 Hermes persona worker
- 事件调度、场景生成与评测逻辑
- 知识成长、决策沉淀、情境召回相关的实验指标
- 多实例对照实验与长期回放分析

不覆盖范围：
- 当前本地优先单人 MVP 的实现细节
- 当前阶段的 UI / 路由 / 数据库 schema 改动要求
- 知识图谱具体表结构的落地任务拆分
- 当前阶段的运维级 SLA、成本优化与生产安全设计

阶段边界：
- 默认前提：至少等知识图谱、决策对象或同等级的上层结构能力已经具备后，再决定是否推进本设计。
- 若用户未明确拍板，本方案保持“记录态”，不进入 plan / status 主线。

## 5. 关键决策

- 决策 A：把这件事定义为“合成成长实验场”，而不是当前产品主线功能。
  - 核心价值是验证长期成长性，而不是扩充当前功能列表。

- 决策 B：未来重点不是单纯“投喂 note”，而是模拟完整知识行为。
  - 事件至少应覆盖 capture、digest、search、reuse、decision logging、reflection / cleanup。

- 决策 C：优先做行为模拟与评测闭环，不优先做模型训练。
  - 短中期更重要的是 persona prompt、持续状态、场景课程化与评估机制。
  - 只有在行为过于理想化、与真实程序员差异过大时，才考虑更重的训练手段。

- 决策 D：实验数据必须具备时间连续性、任务依赖与错误修正轨迹。
  - 如果只是定时随机塞入 note，最多得到一堆笔记，而不会自然长出知识网络与决策路径。

- 决策 E：未来应支持多个 Hermes persona 和多个 DevBrain 实例做对照实验。
  - 既要模拟规范型输入，也要模拟混乱型、复盘型、决策型等不同记录风格。

- 决策 F：实验目标是判断“系统是否会成熟”，不是证明“系统永远有效”。
  - 必须允许实验结果暴露搜索退化、标签爆炸、知识冲突、错误召回等真实问题。

## 6. 未来实验架构草案

### 6.1 核心角色

1. DevBrain Instance
- 被测试对象。
- 每个实例代表一个合成程序员的知识库，或一个待对照的产品版本。

2. Hermes Persona Worker
- 模拟真实程序员行为。
- 具备稳定 persona、技术栈偏好、记录习惯、整理倾向与复盘频率。

3. Scenario Generator
- 生成连续工作情境，而不是离散随机笔记。
- 例如围绕 hydration bug、monorepo 依赖冲突、部署事故、数据库迁移失误等主题形成多轮事件链。

4. Scheduler / Orchestrator
- 负责以固定基线 + 随机扰动 + burst / quiet 周期调度事件。
- “每 20 分钟一次”可作为未来默认基线之一，但不应成为唯一节奏模型。

5. Evaluator / Judge
- 负责计算成长指标、结构指标、召回指标、维护性指标。
- 将实验结果沉淀为可比较的报告，而不是只看 note 数量。

### 6.2 事件类型

未来实验至少应支持以下事件类型：
- Capture：快速记录碎片知识
- Digest：把 Inbox 整理为结构化知识
- Search：在遇到新问题时先搜索旧知识
- Reuse：把旧知识应用到新场景并留下结果
- Decision Logging：记录方案选择、权衡与适用边界
- Reflection / Cleanup：合并重复项、修正标签、升级可信度、标记过时知识

### 6.3 Persona 维度

未来应至少考虑以下程序员画像：
- 速记型：记录多，整理少
- 复盘型：输入不算高频，但整理深入
- 搜索依赖型：经常先搜旧知识
- 决策型：习惯记录 why / tradeoff
- 混乱型：标题、标签、结构都容易发散

### 6.4 部署阶段建议

Phase A：单机实验室
- 1 台云服务器
- 1 个 orchestration service
- 1~3 个 Hermes worker
- 1~3 个 DevBrain 实例
- 1 套日志 / 指标 / 回放面板

Phase B：多实例对照实验
- 实例 A：当前基线
- 实例 B：增强搜索
- 实例 C：增强相关推荐 / 图结构
- 实例 D：增强决策链或规范化治理

默认建议：先证明实验机制可跑，再考虑更重的容器编排、成本优化或复杂云架构。

## 7. 风险与权衡

- 风险 1：如果未来过早推进本实验，容易干扰当前主线，造成范围膨胀。
- 风险 2：如果 Hermes persona 过于理想化，实验会高估 DevBrain 的真实表现。
- 风险 3：如果只模拟新增 note 而不模拟搜索、复用、回访与修正，实验结论会严重失真。
- 风险 4：如果缺少回放与审计链路，将无法解释实验结果是如何形成的。
- 风险 5：多实例实验会引入额外运维成本；在产品机制尚未稳定前，不值得提前重投入。

权衡结论：
- 当前更值得把它保留为未来设计，而不是立即排进开发。
- 等上层知识结构能力真正具备后，再用该实验场验证“是否真的会成长”。

## 8. 验证方式

这份文档当前不要求实现级验证。
未来若用户明确立项，可按下面方式验证：

- 验证项 1：单 persona、单 DevBrain、持续 3~7 天，确认实验基础链路可运行。
- 验证项 2：对比不同 persona 下的知识增长质量，而不是只比输入量。
- 验证项 3：验证搜索、相关推荐、图关系或决策链是否随着时间积累而变得更有用。
- 验证项 4：验证系统是否会出现标签分叉、重复知识堆积、错误召回、低质量 digest 等退化现象。
- 验证项 5：支持事件回放，能追踪每条合成行为的来源、上下文、写入结果与后续影响。

建议未来指标分组：
- Growth：note 数、digested 占比、decision 数、recall event 数
- Structure：关联度、孤立率、重复率、标签分叉率
- Utility：搜索命中后复用率、相关推荐采纳率、重复问题复用率
- Quality：低质量 digest 比例、冲突知识比例、过时知识比例
- Maintenance：inbox 堆积速度、噪声扩散速度、检索退化速度
