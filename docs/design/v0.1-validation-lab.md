## 1. 背景

`docs/design/product-requirements.md` 已明确 DevBrain 首版的目标是跑通「收集 -> 整理 -> 搜索 -> 复用」主闭环。当前仓库实现也已完成这条 MVP 路径，并进入 handoff 后的复验阶段。

但当前最大问题已经不是“功能是否存在”，而是“这套 MVP 是否已经跨过原型可演示到知识系统可信的门槛”。

从当前状态和 code review 结论看，主要不确定性集中在 3 个方面：
- 搜索结果是否真的能支撑未来复用，而不只是多字段 contains 命中。
- `Inbox -> Digested` 的最小结构门槛是否既能拦住低质量整理，又不至于过重。
- 相关推荐是否真的帮助用户扩展和复用历史经验，而不只是详情页上的装饰性模块。

因此，下一阶段应先把“验证”单独定义为一个设计主题：用较轻量、可持续、可复验的方式，判断 v0.1 是否已经具备真实产品价值，以及哪些结论会触发后续技术升级。

## 2. 目标

- 为当前 MVP 建立一套独立的验证设计，而不是继续默认沿着功能堆叠推进。
- 验证搜索、结构化整理门槛、相关推荐三条关键价值链是否成立。
- 借鉴 `docs/design/synthetic-growth-lab.md` 的“连续事件链”思路，但压缩成适用于当前本地单人 MVP 的轻量实验方案。
- 明确一组可复验的判断指标，用来支撑“继续打磨现有 MVP”与“进入技术升级评估”的分界。
- 为下一阶段的搜索演进和 v0.2 / v0.3 设计提供真实输入，而不是建立在未经验证的主观假设上。

## 3. 非目标

- 不是当前迭代的实现计划，不直接拆分代码任务。
- 不在本设计中直接要求落地 SQLite FTS、embedding、知识图谱或 AI 主流程能力。
- 不把 `docs/design/synthetic-growth-lab.md` 提前拉进当前主线，也不要求云上长期实验基础设施。
- 不追求伪精确的数据仪表盘；当前阶段更重要的是建立可信、可复验的判断框架。
- 不以“note 数量增长”作为主要成功信号。

## 4. 方案边界

本设计只覆盖当前 v0.1 MVP 的价值验证，不改动产品边界：
- 覆盖对象：搜索召回、Digested 质量门槛、相关推荐复用价值。
- 覆盖场景：真实自用、任务链式 demo / review、小规模连续事件验证。
- 覆盖时间尺度：以数天到 1~2 周的短周期复验为主，而不是数月级长期成长观察。

本设计不覆盖：
- 新页面、路由、数据库 schema 的直接改动要求。
- 多用户试点、云端部署或正式埋点体系。
- v0.2 / v0.3 的实现拆解。

## 5. 关键决策

- 决策 A：把“验证”定义为独立阶段，而不是默认继续往功能扩展推进。
  - 当前更需要回答“系统是否真的有用”，而不是“还能再加什么”。

- 决策 B：验证的最小单位不是单个页面，而是完整事件链。
  - 事件链至少覆盖 `capture -> digest -> search -> reuse -> reflection`，否则无法判断系统是否会形成可复用知识。

- 决策 C：优先验证“召回与复用价值”，而不是记录量。
  - 如果知识无法被再次找回并复用，再多条目也只是堆积。

- 决策 D：借鉴 `synthetic-growth-lab` 的方法，但只保留“连续事件、场景链、退化暴露”三类思想。
  - 当前不需要 Hermes worker、云端调度和长期多实例对照实验。

- 决策 E：验证结论必须能触发后续决策。
  - 验证不是为了写一份漂亮报告，而是为了决定：继续打磨现有规则、进入搜索升级，还是调整产品门槛。

## 6. 验证方法设计

### 6.1 验证路径

建议采用 3 条互补路径：

1. **真实自用路径**
   - 用真实开发中的报错、命令、配置坑持续录入。
   - 关注系统是否自然进入工作流，而不是用户是否“刻意配合”它。

2. **任务链式 demo / review 路径**
   - 基于一组连续问题链进行复验，例如：依赖冲突 -> 修复方案 -> 二次变体问题 -> 相关推荐扩展。
   - 重点观察搜索和相关推荐在连续上下文中的表现，而不是单次静态检索。

3. **轻量合成场景路径**
   - 借鉴 `synthetic-growth-lab` 的事件设计方式，人工构造少量连续事件链。
   - 事件至少覆盖 Capture、Digest、Search、Reuse、Reflection / Cleanup。
   - 目标不是制造大规模数据，而是更快暴露退化点。

### 6.2 建议场景族

建议至少覆盖以下 4 类场景：

- **重复问题召回场景**
  - 例如再次遇到相似报错或依赖冲突。
  - 目标：验证搜索是否能把过去的解决经验拉回前几条结果。

- **模糊记忆场景**
  - 用户只记得一段错误片段、一个命令或一个技术栈关键词。
  - 目标：验证系统是否支持“记不准也能找回来”。

- **整理质量场景**
  - 同一条 Inbox 是否能被整理成足够有复用价值的 Digested 条目。
  - 目标：验证当前门槛是过松、过紧还是基本合适。

- **联想复用场景**
  - 用户打开一条知识后，是否能通过相关推荐发现额外有价值的相邻经验。
  - 目标：验证相关推荐是否真的帮助知识扩展。

### 6.3 观察节奏

建议按“短链高频复验”推进：
- 每一轮验证聚焦 5~10 条连续事件，不追求一次性做大。
- 每轮结束后记录：命中情况、误召回情况、门槛摩擦点、相关推荐是否被采纳。
- 至少完成 2~3 轮不同主题的事件链后，再做阶段判断。

## 7. 指标与决策门槛

当前阶段以“轻指标 + 强复盘”为主。建议关注以下信号：

### 7.1 搜索召回指标

- **目标条目 Top 5 命中率**
  - 在验证场景中，目标条目或足够接近的候选应较高频出现在前 5 条。
- **首屏可用率**
  - 用户在无需继续翻页的前提下，能在首屏结果中看到可复用候选。
- **模糊关键词容错性**
  - 用户只输入部分记忆时，仍能较高概率找回相关知识。

建议判断：
- 若多轮复验中，Top 5 命中率持续偏低，应优先进入搜索升级评估。
- 若主要失败原因是标签、技术栈、术语别名分叉，应优先做规则与 canonicalization 增强，而不是直接跳 FTS。

### 7.2 Digested 质量指标

- **低质量 digest 比例**
  - 被标记为 `digested` 的条目中，后续回看仍明显缺少复用所需信息的比例。
- **门槛摩擦点**
  - 用户为了过门槛而机械补字段，但并未提升知识质量的情况。
- **整理后价值提升感**
  - 同一条 Inbox 在整理后是否明显更容易被搜索和复用。

建议判断：
- 若低质量 digest 仍频繁出现，说明门槛过松。
- 若用户频繁为了过门槛而补无意义内容，说明门槛过重或字段设计不对。

### 7.3 相关推荐指标

- **相关推荐点击 / 打开率**
  - 用户是否愿意继续点开推荐条目。
- **相关推荐采纳率**
  - 推荐是否真正提供了额外帮助，而不是只被点击后放弃。
- **误召回率**
  - 推荐结果中明显“看起来相关、实际无用”的比例。

建议判断：
- 若相关推荐长期几乎不被打开或复用，说明当前推荐更多是装饰性功能。
- 若误召回集中来自标签和术语不一致，应优先修输入治理与候选集，而不是先加更复杂信号。

## 8. 风险与权衡

- 风险 1：如果验证样本过少，容易把偶然命中误判为产品价值。
  - 权衡：当前不追求大规模数据，但至少要保证多轮、连续、不同主题的事件链复验。

- 风险 2：如果只做 demo 场景，容易高估系统真实可用性。
  - 权衡：必须保留真实自用路径，避免只在“被设计过的数据”上看起来有效。

- 风险 3：如果过早追求量化仪表盘，会把精力花在埋点而不是产品判断上。
  - 权衡：当前阶段以轻量记录和复盘为主，必要时再补最小埋点。

- 风险 4：如果验证与后续决策脱钩，文档价值会快速归零。
  - 权衡：所有观察结论都要指向一个明确动作：继续打磨、技术升级，或重新校准产品门槛。

## 9. 验证方式

当前设计文档本身不要求实现级验证；待用户拍板进入下一阶段后，建议按以下方式执行：

- 验证项 1：完成 2~3 轮不同主题的连续事件链复验。
- 验证项 2：每轮至少覆盖一次 `capture -> digest -> search -> reuse` 完整链路。
- 验证项 3：对每轮记录搜索命中、相关推荐表现、门槛摩擦点与输入分叉情况。
- 验证项 4：基于复验结果判断：
  - 继续打磨现有 MVP；或
  - 进入搜索 / 召回技术升级评估；或
  - 回调 Digested 质量门槛。
