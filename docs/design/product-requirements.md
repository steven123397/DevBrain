DevBrain 已经完成 v0.1 MVP 主闭环，并进入 v0.2 AI Assist Layer 阶段。当前产品已经不再适合继续沿用“首版草案 PRD + 多份未来设想并行存在”的文档结构。

## 2. 产品愿景

DevBrain 是一个面向程序员的本地优先知识系统，用来把真实开发中的碎片经验沉淀为可检索、可关联、可复用的知识资产。

它的终局不应被定义为“复制一个完整大脑”，而更接近：

> 一个以知识为主资产、以证据为可信来源、以情境为召回触发条件的程序员记忆外骨骼。

这意味着 DevBrain 的长期方向不是做一个万能聊天体，而是逐步帮助用户回答：

- 我过去解决过什么问题？
- 哪些知识对象之间存在稳定关系？
- 我过去为什么这样判断？
- 在当前情境下，哪段经验值得被召回？

## 3. 产品定位

一句话定位：

> 一个帮助程序员把问题、方案、流程、坑点与判断依据沉淀为本地可复用知识资产的认知辅助系统。

更具体地说，DevBrain 的定位不是：

- 普通笔记应用
- agent-first 编程环境
- 云端协作知识库
- 以聊天替代结构化整理的 AI 产品
- 一开始就全量采集电脑行为的行为监控系统

它的第一性价值仍然是：**把程序员的真实经验稳定沉淀，并在需要时可靠找回。**

## 4. 目标用户

### 4.1 当前核心用户

- 单人或小规模自我驱动的程序员
- 经常写代码、调试问题、折腾环境、查资料、沉淀经验的人
- 经常遇到“我以前明明解决过这个问题”的开发者
- 希望长期积累技术资产，但不想维护复杂知识系统的人

### 4.2 当前最优先场景

- 报错排查与复盘
- 工具链 / 环境 / 配置坑沉淀
- 命令片段与操作流程记录
- 技术方案与取舍结论整理
- 再次遇到类似问题时的搜索与复用

## 5. 核心产品原则

### 原则 A：主闭环优先

当前主闭环仍是：

`收集 -> 整理 -> 搜索 -> 复用`

未来任何增强层都不能破坏这条主线。

### 原则 B：知识优先于日志堆积

对程序员阶段来说，系统前台的主资产应是知识对象，而不是原始事件流。

### 原则 C：事件提供证据，情境触发召回

长期看，产品应逐步形成这样一组分工：

- 知识对象：回答“我到底记住了什么”
- 事件证据：回答“我凭什么相信它”
- 情境触发：回答“什么时候应该把它拿出来”

### 原则 D：note-first, object-later, context-after

- 当前阶段继续以 Note 为主模型。
- 下一阶段先把知识对象和关系显式化。
- 再下一阶段才进入情境召回与 repo / project 绑定。

### 原则 E：local-first

在单人阶段，本地数据库仍是 canonical source of truth。未来增加 AI、图谱、召回、上下文绑定时，也应优先扩展本地模型，而不是把主存储外包给云端服务或黑盒模型。

### 原则 F：AI 只能做辅助层

AI 可以帮助提取候选、压缩信息、增强召回，但不能替代结构化整理主流程，更不能直接成为知识可信性的唯一来源。

## 6. 明确非目标

在当前主线下，以下方向仍不进入默认范围：

- AI 主流程替代
- 聊天入口替代列表、详情、编辑与结构化整理
- 多端同步、团队协作、权限体系
- 插件系统、浏览器自动剪藏、命令行全自动导入流水线
- 以图谱可视化作为 v0.3 的优先目标
- 在 v0.3 之前就做重型 repo / file 深绑定
- 全量行为采集与“模拟完整大脑”的产品承诺

## 7. 当前主线阶段

### 7.1 v0.1：Knowledge Cards MVP（已完成）

v0.1 已完成主闭环与搜索 / 相关推荐质量升级，相关范围、最终边界与验证结论见 `docs/design/v0.1-mvp-foundation.md`。

### 7.2 v0.2：AI Assist Layer（当前阶段）

阶段目标：

- 降低结构化整理成本
- 提升模糊场景下的召回与判断效率
- 保持 AI 为候选态，不写死主数据

当前能力边界：

- AI 结构字段提取
- AI 标签 / 技术栈建议
- 搜索结果压缩提示
- 无 API key 自动降级

### 7.3 v0.3：Knowledge Object & Relation Layer

阶段目标：

- 把当前以 Note 为中心的知识卡片系统，升级为以知识对象和显式关系为核心的知识系统
- 让“问题 / 方案 / 流程 / 坑点 / 决策”逐步成为可复用对象
- 让“相关”升级为可解释的语义关系

这一阶段不以图谱可视化为优先目标。图只是关系层的一种视图，而不是 v0.3 的核心交付物。

### 7.4 v0.4：Contextual Recall Layer

阶段目标：

- 把项目、仓库、文件、任务、报错等情境逐步作为召回触发器
- 让知识在用户重新进入类似情境时被更自然地激活
- 让决策路径、适用边界与验证证据的复用更加自动化

repo / project / file 绑定属于这一阶段的一部分，但不等于这一阶段的全部。

## 8. 阶段承接关系

### 从 v0.1 到 v0.2

前提：

- 基础主闭环已成立
- 搜索与相关推荐已有基础价值
- 知道 AI 是增强什么，而不是补救什么

### 从 v0.2 到 v0.3

前提：

- AI 辅助结果已帮助降低整理成本或提升模糊召回
- 当前 Note 主模型和搜索底座足够稳定
- 已确认用户确实需要更显式的对象、关系和决策沉淀

### 从 v0.3 到 v0.4

前提：

- 知识对象和关系语义已经稳定
- 决策路径开始形成可复用资产
- 已确认用户需要“情境触发召回”，而不是更多静态知识条目

## 9. 当前成功标准

当前阶段不以“记住更多东西”为主要成功信号，而以这些问题为准：

- 用户是否更容易把碎片经验整理成高质量知识
- 用户是否更容易在模糊记忆场景下找回旧经验
- 用户是否能更快判断一条知识是否值得打开和复用
- 产品是否仍保持本地可信、结构清晰和主线克制
