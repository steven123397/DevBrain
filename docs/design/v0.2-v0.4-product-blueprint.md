## 1. 这份文档解决什么问题

当前仓库已经完成 v0.1，并进入 v0.2。后续阶段如果继续沿用旧的命名和边界，会出现两个明显问题：

- 容易把 v0.3 误解成“开始做图谱可视化”
- 容易把 v0.4 误解成“只是把 repo / file 绑进来”

为了让未来主线与当前已经形成的产品理解一致，需要重新定义 v0.2、v0.3、v0.4 的阶段目标与承接关系。

## 2. 总体路线

DevBrain 的主线现在应理解为：

- v0.1：Knowledge Cards MVP（已完成）
- v0.2：AI Assist Layer（进行中）
- v0.3：Knowledge Object & Relation Layer
- v0.4：Contextual Recall Layer

这条路线背后的统一逻辑是：

1. 先跑通知识卡片主闭环。
2. 再用 AI 降低整理成本、增强模糊召回。
3. 再把知识卡片升级成知识对象与显式关系。
4. 最后让知识在真实情境中被更自然地激活。

## 3. 关键决策

### 决策 A：v0.2 继续定义为 AI Assist Layer

AI 只做辅助层，不做主流程替代。

### 决策 B：v0.3 不再按“图谱版本”理解

v0.3 的核心不是做图，而是把知识对象和关系显式化。图谱浏览只是其中一个可能视图，而不是阶段目标本身。

### 决策 C：v0.4 不再只按“Context & Repo Layer”理解

repo / project / file 绑定只是情境召回的一部分。v0.4 的核心是让知识在用户重新进入类似情境时被更自然地激活，因此更准确的阶段名称应是 `Contextual Recall Layer`。

### 决策 D：程序员阶段的主资产是知识对象

对当前 DevBrain 来说，更合理的长期分工是：

- 知识对象是主资产
- 事件记录是证据来源
- 情境信号是召回触发器

### 决策 E：继续坚持 note-first, graph-later, context-after

- 先用 Note 承载稳定主闭环
- 再抽出对象和关系
- 再引入情境与 repo 绑定

## 4. v0.2：AI Assist Layer

### 4.1 阶段目标

在不破坏现有主流程的前提下，降低结构化整理成本，并增强模糊记忆场景下的召回与判断效率。

### 4.2 核心能力

- 结构字段候选提取
- 标签 / 技术栈建议
- 搜索结果压缩提示
- 无 key 自动降级
- 所有 AI 输出保持候选态

### 4.3 阶段非目标

- 不做 AI-first 工作流
- 不做自动整理并直接写库
- 不做聊天入口替代编辑 / 搜索 / 详情
- 不做重型 embedding / graph / repo 绑定主线

### 4.4 进入下一阶段的前提

- AI 辅助确实降低了整理成本或提升了模糊召回
- 当前 Note 主模型、搜索与相关推荐已经足够稳定
- 已经看到“知识对象化”的真实需求，而不是为了追新概念升级

## 5. v0.3：Knowledge Object & Relation Layer

### 5.1 阶段定义

v0.3 的目标不是把笔记画成图，而是把当前以 Note 为中心的知识卡片系统，升级成以知识对象和显式关系为核心的知识系统。

### 5.2 这一阶段要解决的问题

- 当前系统能存和搜知识，但知识仍主要被包在 Note 里
- 相关推荐大多还是“相关”，而不是真正的语义关系
- “为什么这样选”“适用边界是什么”还没有沉淀成稳定资产
- 用户能找到旧条目，但还不能围绕一个问题展开一串有结构的复用

### 5.3 核心能力

- 从 Note 中逐步显式化 `Problem / Solution / Procedure / Pitfall / Decision / Concept`
- 把“相关”升级为 `solves / depends_on / similar_to / applies_in / avoid_when` 等显式关系
- 让决策路径与适用边界开始成为可复用对象
- 支持围绕某个问题、方案或决策进行更结构化的探索与复用

### 5.4 阶段非目标

- 不把图谱可视化作为优先交付
- 不做大规模 repo / file 深绑定
- 不做全量行为采集
- 不把 AI 生成对象直接当成可信事实
- 不一次性落完整“记忆内核”

### 5.5 与未来数据模型的关系

v0.3 是 `docs/design/future-data-model.md` 的第一阶段落地入口。重点不是把所有未来实体一次落全，而是先证明：

- 知识对象能稳定从 Note 中长出来
- 显式关系确实有复用价值
- 决策与适用边界值得成为一等资产

## 6. v0.4：Contextual Recall Layer

### 6.1 阶段定义

v0.4 的目标是把项目、仓库、文件、任务、报错、时间与复用记录逐步作为情境信号，让知识在类似情境中被更自然地激活。

### 6.2 这一阶段要解决的问题

- 当前系统主要依赖手动搜索和主动打开
- 知识与真实工作情境之间的连接还很弱
- 决策路径、适用边界、验证证据还不能在重新进入某项目时自然复用

### 6.3 核心能力

- 建立 `Project / Repo / File / Context` 等情境锚点
- 让知识与代码对象、任务阶段、错误签名等建立更稳定映射
- 引入基于情境的召回触发与排序增强
- 记录知识被召回后的使用结果，为后续演化和能力判断提供反馈

### 6.4 阶段非目标

- 不做全量行为监控优先
- 不做“自动替用户完成所有工作”的代理系统
- 不把 repo 绑定误当成阶段全部价值

## 7. 阶段承接关系

### v0.2 -> v0.3

前提：

- AI 辅助结果稳定可用
- 搜索与相关推荐底座稳定
- 用户开始显著需要对象化与关系化，而不只是更多候选摘要

### v0.3 -> v0.4

前提：

- 知识对象和关系语义已经足够稳定
- 决策路径与适用边界开始形成可复用资产
- 已确认用户需要的是“在情境中自然想起”，而不是更复杂的静态知识树

## 8. 风险与权衡

- 风险 1：过早把 v0.3 做成图谱工程
  - 权衡：对象和关系语义先于视图

- 风险 2：过早把 v0.4 做成 repo 集成工程
  - 权衡：情境召回先于大规模上下文接入

- 风险 3：让未来设计反向干扰当前 v0.2 收口
  - 权衡：阶段定义清晰，但不提前抢跑实现
