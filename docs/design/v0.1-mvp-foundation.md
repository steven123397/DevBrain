## 2. v0.1 的核心目标

v0.1 的目标已经很明确，并且现已全部完成：

1. 跑通 `收集 -> 整理 -> 搜索 -> 复用` 主闭环。
2. 让知识条目具备结构化表达，而不是只存自由文本。
3. 支持本地搜索、标签 / 技术栈筛选与基础相关推荐。
4. 支持从 Inbox 碎片条目整理为 Digested 正式条目。
5. 在不依赖 AI 的前提下，证明产品具备基础复用价值。

## 3. v0.1 的最终边界

### 3.1 产品范围

v0.1 交付的是一个面向程序员的本地优先知识卡片系统。

最终范围包括：

- 单人、本地优先
- Inbox -> Digested 工作流
- SQLite 持久化
- 搜索、筛选、详情页
- 规则化相关推荐
- demo seed、本地 review 路径、删除流程、空状态处理

### 3.2 明确非目标

v0.1 明确不做：

- AI 主流程
- embedding 检索
- 知识图谱可视化
- 多端同步、团队协作、权限控制
- 浏览器插件、自动抓取、自动导入流水线
- 云端服务依赖

## 4. v0.1 的最终信息架构

### 页面结构

1. `/`
   - 仪表盘
   - 最近新增 / 最近更新
   - Inbox / Digested 数量
   - 快速新建入口

2. `/notes`
   - 搜索框
   - 状态筛选
   - 标签筛选
   - 技术栈筛选
   - 排序

3. `/notes/new`
   - 新建条目
   - 快速录入为主

4. `/notes/[id]`
   - 条目详情
   - 结构化内容
   - 相关推荐

5. `/notes/[id]/edit`
   - 编辑与整理
   - Inbox -> Digested 转化

### 主闭环

1. 快速记录碎片到 Inbox
2. 通过编辑页整理为 Digested
3. 通过搜索、筛选与相关推荐找回旧经验
4. 在类似问题中复用历史知识

## 5. v0.1 的数据基线

v0.1 保持最小而稳定的数据模型：

- `Note`
  - 标题、原始输入、摘要、问题、方案、why、命令、参考、状态、可信度、技术栈、来源
- `Tag`
- `NoteTag`

设计结论：

- 继续以 `Note` 为主模型完全正确
- 不在 v0.1 就引入显式知识对象与关系边
- 相关推荐先按规则动态计算，不做独立关系持久化

## 6. v0.1 的关键设计取舍

### 取舍 A：先做结构化知识卡片，不做聊天式系统

原因：

- 首版要证明的是知识沉淀与召回价值
- 聊天入口会稀释结构化主线

### 取舍 B：先做规则搜索与相关推荐，不急于上 FTS / embedding

原因：

- v0.1 首先需要一套可解释、可调试、可验证的基线
- 当时更大的问题是输入分叉和结构质量，而不是检索引擎不够“聪明”

### 取舍 C：Inbox 与 Digested 分层

原因：

- 程序员在踩坑现场需要低负担记录
- 真正高质量的知识整理可以延后发生

### 取舍 D：AI 不是首版前提

原因：

- 首版必须在纯本地下完整可用
- 否则无法证明产品基础价值

## 7. v0.1 的验证主题

v0.1 最后真正需要回答的不是“功能是否存在”，而是“这套 MVP 是否已经跨过原型可演示到知识系统可信的门槛”。

最终验证重点集中在 3 条价值链：

### 7.1 搜索召回

要验证：

- 模糊关键词下是否能找回目标条目或近似条目
- 输入分叉是否明显侵蚀搜索质量
- 搜索首屏是否具备复用价值

### 7.2 Digested 质量门槛

要验证：

- `Inbox -> Digested` 是否真的提高了知识质量
- 当前最小门槛是否过松或过重
- 用户是否是在“为过门槛而补字段”，还是确实获得了更高复用价值

### 7.3 相关推荐价值

要验证：

- 推荐是否帮助用户扩展到相邻经验
- 推荐是不是只是详情页上的装饰性模块
- 当前误召回更像候选集问题、输入治理问题，还是排序问题

## 8. v0.1 的最终验证结论

结合当前仓库状态与后续完成记录，可以认为 v0.1 已经过关，结论如下。

### 8.1 搜索主路径成立

- 多字段关键词召回、URL 持久化、标签 / 技术栈辅助召回已经稳定可用
- 经过一轮搜索 / 相关推荐质量升级后，当前结论仍是“规则层增强优先，暂不急于进入 SQLite FTS”

### 8.2 Digested 门槛成立

- `summary + problem + solution` 加上标签 / 技术栈的最小门槛，已经能明显区分整理前后价值
- 目前没有证据表明应立即放松或大幅提高门槛

### 8.3 相关推荐主路径成立，但仍是规则基线

- 相关推荐已从装饰性模块提升为有实际复用价值的辅助模块
- 当前仍停留在规则化相关推荐阶段，后续应优先增强语义结构而不是先做可视化

### 8.4 v0.1 的产品判断

v0.1 可以被视为“已完成并成立”的基础版本：

- 主闭环已跑通
- 搜索与相关推荐已完成一轮真实验证
- 结构化整理门槛已具备基础可用性
- 产品可以进入 v0.2，而不是继续停留在 MVP 生死验证阶段

## 9. v0.1 关闭后的承接关系

v0.1 完成后，后续主线不再是继续堆叠 MVP 功能，而是进入：

- v0.2：AI Assist Layer
- v0.3：Knowledge Object & Relation Layer
- v0.4：Contextual Recall Layer

这也意味着：

- 本文不再继续追加新的活跃设计
- v0.1 的新增实现记录只保留在 `docs/plan/history.md` 与 `docs/status/project-status.md`
- 后续阶段设计统一回到主 PRD 与新的阶段蓝图文档
