# DevBrain 搜索与召回演进设计

负责人：Hermes

## 1. 背景

当前 v0.1 已具备可运行的搜索、筛选与相关推荐闭环：
- 搜索基于多字段关键词召回。
- 结果集已补显式相关性排序。
- 标签与技术栈已补最小 canonicalization。
- 相关推荐仍采用规则驱动、可解释优先的方案。

这一基线足以支撑 MVP，但尚不能直接推导出后续演进路线。当前真正需要回答的问题不是“要不要做更高级的搜索”，而是：
- 当前方案还能支撑到什么规模和复杂度。
- 退化主要来自检索机制本身，还是来自输入分叉与候选质量。
- 什么时候值得引入 SQLite FTS。
- v0.2 的 AI Assist 应如何建立在当前召回基础之上，而不是把脆弱底座用 AI 包起来。

因此，需要单独定义一份搜索与召回演进设计，给下一阶段的升级判断建立边界和门槛。

## 2. 目标

- 明确当前搜索与相关推荐基线的适用边界。
- 把演进路径拆成多个层级，而不是直接把“升级”理解成上 FTS 或 embedding。
- 为搜索退化、相关推荐失真、输入治理不足等不同问题建立不同的升级路径。
- 明确什么现象优先触发规则增强，什么现象才触发 SQLite FTS 评估。
- 为 v0.2 的 AI Assist 和更长期的知识网络能力预留稳定演进缝隙。

## 3. 非目标

- 不是当前迭代的实现清单，不直接要求修改代码或数据库 schema。
- 不把 SQLite FTS 视为默认必做项。
- 不把 embedding 或 AI 召回提前变成当前产品可用性的前提。
- 不在本设计中讨论知识图谱对象建模细节；那部分由 `docs/design/future-data-model.md` 和后续版本蓝图负责。
- 不追求“不透明但看起来更聪明”的黑盒召回系统。

## 4. 方案边界

本设计只覆盖搜索、筛选、相关推荐及其所依赖的最小输入治理：
- 覆盖：关键词召回、排序信号、标签 / 技术栈治理、相关推荐候选与排序。
- 关联但不展开：v0.2 的 embedding、压缩提示、AI 辅助提取。
- 不覆盖：多用户检索、云端搜索服务、复杂权限隔离、知识图谱边表实现。

## 5. 关键决策

- 决策 A：保留“规则优先、可解释优先”的阶段性原则。
  - 在 v0.1 到 v0.2 过渡前，搜索与相关推荐仍应尽量保持可解释，而不是用更复杂机制掩盖问题。

- 决策 B：把演进路线拆成 3 层，而不是一步跳到 FTS 或 AI。
  - 层 1：规则增强与输入治理。
  - 层 2：SQLite FTS。
  - 层 3：AI / embedding 增强召回。

- 决策 C：FTS 的触发条件应来自验证结论，而不是工程冲动。
  - 只有当当前关键词召回在质量或规模上持续退化，且规则增强已不足时，才进入 FTS 评估。

- 决策 D：相关推荐优先修“候选质量”，再谈更复杂排序信号。
  - 如果候选集合本身就偏离真实有用知识，加再多分数权重也只是在优化错误集合。

- 决策 E：输入治理是搜索和相关推荐的共同底座。
  - 标签、技术栈、常见术语、标题风格的分叉问题，会同时侵蚀搜索与相关推荐质量。

## 6. 当前基线与主要问题

### 6.1 当前基线

当前方案可概括为：
- 搜索：多字段关键词匹配 + 结果集加权排序。
- 筛选：状态、标签、技术栈、排序参数。
- 相关推荐：共享标签、同技术栈、标题词和命令词重叠。
- 输入治理：最小 canonicalization，降低常见别名分叉。

### 6.2 当前主要问题

当前风险主要有 4 类：

1. **数据增长后的召回退化**
   - 条目变多后，多字段 contains 方案可能变得更难调优。

2. **输入分叉侵蚀质量**
   - 同一概念被多种标签、术语、标题风格表达，会让搜索和相关推荐同时失真。

3. **相关推荐信号较弱**
   - 共享标签和词重叠可解释，但不一定稳定代表“真正值得复用”。

4. **升级决策容易失焦**
   - 如果没有门槛定义，团队可能会把任何问题都解释成“该上 FTS”或“该上 AI”。

## 7. 演进路线

### 7.1 层 1：规则增强与输入治理

这是默认优先层。

可考虑的增强方向：
- 扩展标签 / 技术栈 / 常见术语的 alias 治理。
- 强化标题与摘要中的关键词归一化。
- 为相关推荐增加更稳的候选缩窄规则，例如优先限定同主题、同栈或同问题簇。
- 在排序中继续补“结构字段优先级”，避免原始噪声字段过度干扰。
- 对低质量 Digested 条目设置更稳的排序惩罚，减少噪声条目误入前列。

适用场景：
- 主要问题来自输入不一致、候选集太噪、规则偏弱。
- 数据规模尚小，性能不是首要瓶颈。
- 用户更需要可解释和可调试的结果。

### 7.2 层 2：SQLite FTS

这是面向数据增长后的检索升级层。

适合解决的问题：
- 当前关键词召回在数据量增长后明显退化。
- 多字段 `LIKE` 匹配已难以兼顾质量与性能。
- 查询延迟、排序调优、检索精度开始成为持续问题。

引入 FTS 后的阶段目标：
- 提高检索效率和基础召回能力。
- 保留与现有 `note.search.ts` / service seam 的兼容演进路径。
- 让搜索升级成为结构化增强，而不是重写全部搜索逻辑。

但 FTS 不是万能解：
- 它不能自动修复输入分叉。
- 它不能替代相关推荐的候选质量控制。
- 它不能替代结构化质量治理。

### 7.3 层 3：AI / embedding 增强召回

这是 v0.2 之后才值得进入的增强层。

典型用途：
- 处理关键词不完全重叠但语义接近的条目。
- 为搜索结果补相似候选和压缩提示。
- 帮助用户在“我记不清关键词，但大概知道问题语义”时更快召回。

前提条件：
- 基础关键词召回已基本可用。
- 输入质量没有明显失控。
- 相关推荐和搜索的核心误差已知，否则 AI 层只会把底层问题变得更难解释。

## 8. 升级决策门槛

### 8.1 优先做规则增强的情况

如果验证中主要观察到以下问题，应优先留在层 1：
- 搜索 miss 多来自术语别名、标签分叉、标题表达不一致。
- 相关推荐误召回多来自候选过宽，而不是排序分不准。
- 数据规模不大，但结果波动较大。
- 用户仍需要能解释“为什么搜到它 / 为什么推荐它”。

### 8.2 优先评估 SQLite FTS 的情况

如果验证中出现以下现象，应进入层 2 评估：
- Top 5 命中率和首屏可用率在多轮验证中持续偏低，且规则增强难以改善。
- 数据规模增长后，查询性能和调优复杂度明显上升。
- 当前多字段 contains 逻辑已经难以表达所需的检索语义。

### 8.3 优先增强相关推荐的情况

如果验证中搜索基本可用，但相关推荐长期不被采用，应优先处理：
- 候选集缩窄。
- 输入治理。
- 结构字段与上下文信号的权重。

而不应直接跳到更复杂的黑盒排序。

### 8.4 进入 AI / embedding 评估的情况

只有当以下前提基本满足时，才值得评估层 3：
- 关键词搜索已足够可靠。
- 用户开始明显遇到“语义接近但关键词不重叠”的召回瓶颈。
- 当前产品希望提升的是“模糊记忆召回”和“跨表述方式连接”，而不是补救底层质量问题。

## 9. 风险与权衡

- 风险 1：把所有问题都解释成“搜索引擎不够强”。
  - 权衡：先分清是检索退化、输入分叉还是候选质量问题。

- 风险 2：过早引入更复杂能力，导致系统更难解释和调试。
  - 权衡：在 v0.1 到 v0.2 的过渡期，坚持可解释优先。

- 风险 3：把 FTS 当成一劳永逸的答案。
  - 权衡：FTS 解决的是一类问题，不会自动解决知识质量治理。

- 风险 4：把 AI / embedding 提前当成可用性补丁。
  - 权衡：AI 只能增强稳定底座，不能替代底座本身。

## 10. 验证方式

当前设计文档本身不要求实现级验证；待用户拍板进入下一阶段后，建议按以下方式判断升级路径：

- 验证项 1：基于 `docs/design/v0.1-validation-lab.md` 的多轮复验结果，区分问题来源。
- 验证项 2：记录搜索 miss、相关推荐误召回与输入分叉的主要模式。
- 验证项 3：只有在规则增强难以继续改善且数据规模 / 查询复杂度确实上升时，才进入 FTS 计划设计。
- 验证项 4：只有在基础召回稳定后，才把 embedding / AI 增强提上优先级。
