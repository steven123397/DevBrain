## 1. 背景

当前仓库已经具备可运行的搜索、筛选与相关推荐闭环：

- 搜索基于多字段关键词召回
- 结果集已补显式相关性排序
- 标签与技术栈已补最小 canonicalization
- 相关推荐仍采用规则驱动、可解释优先的方案

这一基线足以支撑 v0.1 / v0.2，但不等于后续路线已经自然清楚。当前真正需要回答的问题不是“要不要上更高级的搜索”，而是：

- 当前方案还能支撑到什么规模和复杂度
- 退化主要来自检索机制、输入分叉，还是候选质量
- 什么时候值得引入 SQLite FTS
- AI / embedding 应该增强什么，而不是掩盖什么
- 后续知识对象与关系层应该怎样建立在稳定召回底座之上

## 2. 目标

- 明确当前搜索与相关推荐基线的适用边界
- 把升级路径拆成多个层级，而不是直接把“升级”理解成上 FTS 或 embedding
- 区分检索退化、输入分叉、候选失真与关系语义缺失
- 为 v0.2、v0.3、v0.4 预留稳定的召回演进缝隙

## 3. 非目标

- 不是当前实现计划，不直接要求修改代码或数据库 schema
- 不把 SQLite FTS 视为默认必做项
- 不把 embedding 或 AI 召回提前变成当前产品可用性的前提
- 不在本设计中展开知识对象建模细节；那部分由 `docs/design/future-data-model.md` 负责
- 不追求不透明但“看起来更聪明”的黑盒召回系统

## 4. 当前基线

当前方案可概括为：

- 搜索：多字段关键词匹配 + 结果集加权排序
- 筛选：状态、标签、技术栈、排序参数
- 相关推荐：共享标签、同技术栈、标题词和命令词重叠
- 输入治理：最小 canonicalization，降低常见别名分叉

## 5. 当前问题类型

### 5.1 检索退化

条目变多后，多字段 contains 方案会越来越难调优。

### 5.2 输入分叉

同一概念被多种标签、术语和标题风格表达，会同时侵蚀搜索和相关推荐质量。

### 5.3 相关推荐信号偏弱

共享标签和词重叠虽然可解释，但不一定稳定代表“真正值得复用”。

### 5.4 关系语义缺失

在 v0.3 之前，系统更多只能表达“相关”，还不能稳定表达“为什么相关”。

## 6. 演进路线

### 层 1：规则增强与输入治理

这是默认优先层。

适合解决：

- 术语别名、标签分叉、标题表达不一致
- 相关推荐候选太宽、太噪
- 结构字段排序信号还不够稳

### 层 2：SQLite FTS

这是面向数据增长后的检索升级层。

适合解决：

- 多字段 `LIKE` 难以兼顾质量与性能
- 查询延迟、排序调优、检索精度开始成为持续问题

### 层 3：AI / embedding 增强召回

这是 v0.2 之后才值得评估的增强层。

适合解决：

- 关键词不完全重叠但语义接近的条目召回
- 搜索结果压缩提示
- 模糊记忆场景下的候选补强

### 层 4：对象与关系增强召回

这是更贴近 v0.3 的方向。

适合解决：

- 系统需要表达 `Problem -> Solution -> Procedure -> Decision` 的链式复用
- 用户需要的不只是“相似条目”，而是“结构化相邻知识”

### 层 5：情境触发召回

这是 v0.4 的方向。

适合解决：

- 用户重新进入某个 repo / 项目 / 任务 / 错误情境时，希望知识被更自然地激活

## 7. 升级决策门槛

### 7.1 优先做规则增强的情况

- 搜索 miss 主要来自术语别名、标签分叉、标题不一致
- 相关推荐误召回主要来自候选过宽
- 数据规模尚小，但结果波动较大

### 7.2 优先评估 SQLite FTS 的情况

- Top 5 命中率与首屏可用率持续偏低
- 数据规模增长后，查询性能和调优复杂度明显上升
- 当前多字段 contains 已难以表达所需的检索语义

### 7.3 优先增强对象与关系的情况

- 搜索基本可用，但“结构化复用”仍明显不足
- 用户需要围绕问题、方案、流程、决策展开一串知识，而不是只要更多相似条目

### 7.4 优先进入 AI / embedding 评估的情况

- 基础关键词召回已基本稳定
- 用户开始明显遇到“语义接近但关键词不重叠”的召回瓶颈
- 当前目标是在稳定底座上增强，而不是补救底座本身

## 8. 风险与权衡

- 风险 1：把所有问题都解释成“搜索引擎不够强”
  - 权衡：先分清是检索退化、输入分叉、候选质量还是关系语义问题

- 风险 2：过早引入更复杂能力，导致系统更难解释和调试
  - 权衡：在 v0.2 到 v0.3 过渡期继续坚持可解释优先

- 风险 3：把 FTS 当成一劳永逸的答案
  - 权衡：FTS 解决的是一类检索问题，不会自动解决知识质量治理

- 风险 4：把 AI / embedding 提前当成可用性补丁
  - 权衡：AI 只能增强稳定底座，不能替代稳定底座

## 9. 验证方式

当前设计文档本身不要求实现级验证；待用户拍板进入下一阶段后，建议按以下方式判断升级路径：

- 基于 `docs/design/v0.1-mvp-foundation.md` 的完成结论确认当前基线是否已成立
- 记录搜索 miss、相关推荐误召回、结构化复用不足与输入分叉的主要模式
- 只有在规则增强难以继续改善且数据规模 / 查询复杂度确实上升时，才进入 FTS 计划设计
- 只有在基础召回稳定后，才把 embedding / AI 增强提上优先级
- 只有在用户开始显著需要“对象链路复用”时，才进入对象与关系增强阶段
