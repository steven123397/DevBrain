## 1. 目标与完成定义

- 目标：在不改变 v0.1 主流程（收集 → 整理 → 搜索 → 复用）的前提下，为知识卡片系统增加 AI 辅助层，降低结构化整理成本、提升模糊场景下的召回与复用效率。
- 完成定义：
  1. 用户在编辑页可一键获取 AI 提取的 `problem / solution / why / summary` 候选，并可逐字段接受、修改或拒绝。
  2. 用户在编辑页可获取 AI 建议的标签与技术栈候选，并可选择性采纳。
  3. 搜索结果列表可展示 AI 生成的压缩摘要提示，帮助用户更快判断条目价值。
  4. 未配置 AI API key 时，系统全部功能正常可用，AI 辅助入口优雅降级为不可见或禁用状态。
  5. 所有 AI 结果均为候选态，不自动写入主数据，必须经用户确认后才持久化。

## 2. 范围

- In scope：
  - AI provider 抽象层（支持至少一个 LLM provider，预留扩展缝）
  - API key 与 AI 功能开关的配置机制
  - 从 `raw_input` 提取结构化字段候选（problem / solution / why / summary）
  - AI 标签与技术栈建议
  - 编辑页 AI 候选展示与接受/拒绝交互
  - 搜索结果压缩提示
  - 无 API key 时的优雅降级
  - 对应单元测试与 E2E 覆盖

- Out of scope：
  - Embedding 向量存储与相似召回（视 Phase B 验证结果决定是否在本轮启动）
  - AI 聊天入口或对话式交互
  - 自动整理（跳过用户审阅直接写入）
  - 知识图谱对象与显式关系（v0.3 范围）
  - 多 provider 同时调用、模型切换 UI
  - 云端服务依赖或用户账号体系

## 6. 风险与依赖

- 风险 1：AI provider API 调用延迟影响编辑体验
  - 缓解：所有 AI 调用均为用户主动触发 + 异步加载，不阻塞表单操作；UI 展示 loading 态

- 风险 2：prompt 质量不稳定导致候选字段可用性波动
  - 缓解：prompt 模板集中管理在 `src/features/ai/prompts/`，便于迭代调优；候选附带 confidence 标记

- 风险 3：AI 候选 UI 侵入编辑主流程，增加非 AI 用户的认知负担
  - 缓解：AI 入口在无 key 时完全隐藏；有 key 时也不自动弹出，需用户主动触发

- 风险 4：百炼模型调用费用与速率限制
  - 缓解：提取与建议为单次请求、非流式；压缩提示采用批量接口减少调用次数

- 依赖 1：`feat/bootstrap-app-shell` 分支合并到 `main`（Task A1）
- 依赖 2：至少一个可用的百炼 API key（开发阶段可用 mock provider 替代）
- 依赖 3：可访问百炼 Anthropic 兼容接口的网络环境与模型配额
