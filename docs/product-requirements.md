# DevBrain 产品需求文档（PRD）

文档状态：v0.1 草案
最后更新：2026-04-11
项目路径：`/home/liangjiaqi/projects/devbrain`
角色分工：Hermes 负责产品/规划/验收口径；Codex 负责代码实现；用户负责方向拍板与关键取舍。

## 1. 产品愿景

DevBrain 是一个面向程序员的本地优先知识库，用来吸收、消化并召回日常开发中遇到的碎片知识：报错、配置坑、调试结论、命令片段、设计判断、踩坑复盘，以及它们之间的关联。

它不是普通笔记应用，也不是一开始就做成“万能 AI 大脑”。
首版目标更务实：

1. 让程序员可以低摩擦地把碎片知识收进来。
2. 让碎片知识可以被整理成结构化知识卡片。
3. 让这些知识在下一次遇到类似问题时能被快速找回。

### 长期北极星

DevBrain 的终局不是“更聪明的笔记本”，而是一个不断生长的个人知识网络与情境召回系统。

它的演进方向应当是：
- 从碎片记录，长成结构化知识卡片
- 从知识卡片，长成可解释的知识网
- 从知识网，长成带有情境触发能力的召回系统
- 从召回系统，逐步辅助用户沉淀决策路径、识别知识空白与邻近成长方向

换句话说，DevBrain 未来不只回答“我记过什么”，还要逐步帮助回答：
- 这些知识之间有什么关系
- 我在什么情境下该调用哪段经验
- 我过去是如何做判断的
- 我当前知识体系里哪些区域密、哪些区域稀、哪些区域正在生长

## 2. 背景与问题定义

程序员每天会遇到大量零碎但真实有价值的信息：
- 一个古怪报错的根因
- 某个环境变量的正确写法
- 某个工具链在特定版本下的兼容性问题
- 一条有效但不容易记住的终端命令
- 一次排查后得出的设计结论

这些信息通常有几个共同问题：
- 当下解决了，但没有沉淀
- 即使记了，也只是散落在聊天、终端、网页收藏、临时笔记里
- 下次搜索时，不知道该用什么关键词
- 同类问题之间没有联系，知识不会“长成体系”
- 重复踩坑，浪费时间与注意力

DevBrain 要解决的问题是：
“如何把程序员的碎片经验，持续转化为未来可复用的能力。”

## 3. 产品定位

一句话定位：
一个帮助程序员把踩坑记录、解决方案与思考结论沉淀成可检索、可关联、可复用知识资产的本地第二大脑。

边界声明：
- 首版是单人使用的本地优先产品。
- 首版优先做知识沉淀与召回，不做团队协作。
- 首版以结构化知识卡片为中心，不以聊天代理为中心。
- 知识图谱、语义召回、自动化采集属于后续增强层，不是 v0.1 的交付前提。

## 4. 目标用户

### 4.1 核心用户
经常写代码、折腾环境、调试问题、查资料、总结技巧的个人开发者。

### 4.2 首版最适配用户画像
- 单兵开发者
- 有记录习惯但知识长期分散的人
- 会频繁遇到“我以前明明解决过这个问题”的程序员
- 希望构建长期技术资产，但不想维护复杂知识系统的人

## 5. 产品目标与非目标

### 5.1 v0.1 目标
1. 跑通“收集 -> 整理 -> 搜索 -> 复用”主闭环。
2. 支持知识条目的结构化表达，而非只存大段自由文本。
3. 支持本地全文搜索与标签/技术栈筛选。
4. 支持从 Inbox 碎片条目整理为 Digested 正式条目。
5. 在条目详情页展示规则驱动的“相关条目推荐”。

### 5.2 v0.1 非目标
1. 不做团队协作、分享、评论、权限控制。
2. 不做多端同步。
3. 不做浏览器插件、命令行自动抓取、网页一键剪藏。
4. 不做真正的图数据库和复杂知识图谱引擎。
5. 不让 AI 成为核心依赖；没有 AI 也必须能完整工作。

## 6. 核心产品原则

### 原则 A：先收进来，再慢慢整理
程序员在遇到问题时，不应该被复杂表单打断。输入必须分层：
- 快速收集：低负担，先存再说
- 深度整理：晚点补充问题、原因、方案和标签

### 原则 B：结构化优先于堆文本
系统必须把“问题、根因、方案、命令、标签、技术栈、来源、可信度”作为一等字段。

### 原则 C：召回价值高于记录数量
判断系统是否有价值，不是看存了多少条，而是看遇到问题时能不能迅速找到并复用。

### 原则 D：底层先稳，智能后置
先把数据模型、检索、状态流转、关系规则做好，再逐步扩到 embedding、知识图谱和 agent 能力。

### 原则 E：情境召回应高于泛化聊天
系统未来最重要的价值，不是像聊天机器人一样“什么都能说”，而是在用户重新进入类似项目、报错、任务或技术情境时，能召回过去真实有效的知识、决策与经验。

## 7. 关键使用场景

### 场景 1：临时踩坑快速收集
用户刚解决完一个问题，希望先快速写一句话或贴一段报错，把记录塞进 Inbox，稍后再补充。

成功标准：30 秒内可完成记录。

### 场景 2：把碎片整理成正式知识
用户有空时，把某条 Inbox 记录补全为正式知识卡片：问题、原因、方案、标签、技术栈、来源、可信度。

成功标准：能明显看出“整理前”和“整理后”的价值差异。

### 场景 3：遇到类似问题时搜索召回
用户遇到报错或模糊记得以前见过某问题，希望通过关键词、标签或技术栈快速找回历史记录。

成功标准：在几秒内找到正确条目或接近条目。

### 场景 4：通过条目联想找到相关知识
用户点开一条知识时，能看到同技术栈、同标签、相似问题、相似命令等相关条目。

成功标准：帮助用户从一条知识扩展到一串相关经验。

## 8. MVP 功能范围

### 8.1 必做功能

#### F1. 新建知识条目
支持新建一条知识记录，最少字段：
- 标题
- 原始输入 raw_input
- 状态 status（默认 inbox）
- 标签 tags（可选）
- 技术栈 stack（可选）
- 来源 source（可选）

#### F2. 条目结构化整理
用户可将某条记录补充为更完整的知识卡片，字段包括：
- problem：问题描述
- solution：解决方案
- why：根因/原理
- summary：提炼后的摘要
- commands：相关命令片段
- references：相关链接/文档引用
- confidence：draft / tested / trusted

#### F3. 条目列表与筛选
支持：
- 最近新增
- 最近更新
- 仅看 inbox
- 仅看 digested
- 按标签筛选
- 按技术栈筛选

#### F4. 搜索
支持：
- 标题搜索
- 原始输入搜索
- problem / solution / why 全文搜索
- 标签辅助召回

#### F5. 条目详情页
展示完整条目内容及元数据：
- 标题与状态
- 核心字段
- 标签 / 技术栈 / 来源
- 创建时间 / 更新时间
- 相关条目推荐

#### F6. 相关条目推荐（规则版）
先不做 embedding；v0.1 使用规则驱动推荐：
- 相同标签
- 相同技术栈
- 标题关键词重叠
- 命令片段关键词重叠
- 最近访问关系加权（可选）

#### F7. Inbox -> Digested 工作流
支持将条目从 `inbox` 转为 `digested`，并在 UI 中明确体现。

### 8.2 可选增强功能（有余力再做）
- AI 自动摘要
- AI 标签建议
- AI 问题/方案字段拆分建议
- 收藏 / 置顶
- markdown 渲染增强

## 9. 信息架构与页面建议

首版建议页面结构如下：

1. `/`
   首页 / 仪表盘
   - 最近新增
   - 最近更新
   - inbox 数量
   - digested 数量
   - 快速新建入口

2. `/notes`
   条目列表页
   - 搜索框
   - 状态筛选
   - 标签筛选
   - 技术栈筛选
   - 排序（最近更新 / 最近创建）

3. `/notes/new`
   新建条目页
   - 快速模式
   - 完整模式

4. `/notes/[id]`
   条目详情页
   - 基础信息
   - 结构化内容
   - 相关条目
   - 编辑按钮

5. `/notes/[id]/edit`
   条目编辑页
   - 从 Inbox 补全为 Digested

## 11. AI 与知识图谱策略

### 11.1 v0.1
AI 不是必须。系统即使在纯本地、无模型能力下也能完整工作。

### 11.2 v0.2
增加 AI Assist Layer，但 AI 角色应以“结构化提取器”和“召回增强器”为主，而不是主流程替代者。

可以考虑加入：
- AI 自动摘要
- AI 标签建议
- AI 从 raw_input 中提取 problem / solution / why 候选字段
- 基于 embedding 的相似条目召回
- 对召回结果生成压缩提示，帮助快速复用历史经验

### 11.3 v0.3+
在底层稳定后，再从“知识卡片系统”扩展到“知识网层”。

优先考虑：
- 显式知识关系边
- 知识图谱视图
- 主题聚类
- 主动复习与关联发现
- 导入外部来源（聊天、终端、网页片段）

这一阶段应开始把 Note 之外的知识对象逐步显式化，例如：
- Concept（概念）
- Problem（问题）
- Solution（方案）
- Procedure（流程）
- Decision（决策）
- Context（上下文）

关系也不应只停留在“相关”，而应逐步支持：
- solves
- depends_on
- similar_to
- used_in
- avoid_when
- derived_from

### 11.4 v0.4+
在知识网层稳定后，再考虑“情境召回层”。

这一阶段的目标是：
- 接入项目 / 仓库 / 文件等上下文
- 让知识与真实代码对象建立映射
- 在用户重新进入类似情境时触发召回，而不只依赖手动搜索
- 逐步沉淀决策路径、适用边界和验证证据

长期可以进一步探索：
- 已掌握 / 半掌握 / 空白区辅助识别
- 邻近知识区域发现
- 基于复用记录的能力地图

原则：
先把“知识主表 + 检索 + 状态流转”做好，再建上层建筑。

## 12. 成功指标

v0.1 不追求复杂埋点，优先看产品可用性指标：

1. 录入成本
- 用户能在 30 秒内创建一条 inbox 记录

2. 整理效率
- 用户能在 3 分钟内把一条 inbox 记录整理成 digested 条目

3. 召回效率
- 用户输入模糊关键词时，前 10 条结果里能较高概率出现目标条目或近似条目

4. 复用价值
- 用户能通过“相关条目”找到额外可复用的历史经验

## 13. 验收标准

若 v0.1 满足以下条件，可视为首版成立：

1. 用户能创建、查看、编辑、删除知识条目。
2. 用户能将条目状态从 inbox 变为 digested。
3. 用户能通过关键词、标签、技术栈找到历史条目。
4. 用户打开某条条目时，能看到合理的相关条目推荐。
5. 页面在空数据、单条数据、多条数据场景下都可正常工作。
6. 本地数据库可稳定保存并读取数据。

## 14. 风险与控制

### 风险 1：过早追求“智能”导致底层空心
控制：把 schema、搜索、编辑流转、详情页作为硬门槛，AI 能力延后。

### 风险 2：字段设计过重，导致录入成本太高
控制：默认支持快速模式；只要求最少字段即可存入 Inbox。

### 风险 3：搜索体验不佳，影响核心价值
控制：v0.1 直接上 SQLite FTS 或等价全文搜索能力，不要只做简单 `LIKE`。

### 风险 4：关系推荐效果太弱
控制：先做可解释的规则推荐；等底层稳定后再换更强算法。

## 15. 版本路线图

### v0.1：Knowledge Cards MVP
- 本地优先 Web App
- SQLite 持久化
- 结构化知识条目
- Inbox / Digested 工作流
- 搜索 / 筛选 / 详情 / 规则推荐

### v0.2：AI Assist Layer
- 自动摘要
- 标签建议
- 问题/方案字段辅助拆分
- embedding 相似搜索

### v0.3：Knowledge Graph Layer
- 显式关系边
- 图谱浏览
- 聚类 / 主题地图
- 主动复习和知识连接建议

### v0.4：Context & Repo Layer
- 项目 / 仓库 / 文件上下文绑定
- 知识与代码对象建立映射
- 基于情境的召回排序
- 让“重新遇到类似情况时快速想起”成为核心体验

### v0.5+：Capability Map / Second Brain Layer
- 决策路径沉淀
- 已掌握 / 半掌握 / 空白区辅助识别
- 邻近知识区域与成长建议
- 更稳定的主动复习与拓展提示

补充说明：
更细的长期实体与关系草案，见 `docs/future-data-model.md`。

## 16. 对 Codex 与 Hermes 的协作要求

### Codex
- 优先完成底层数据模型、存储、搜索和状态流转。
- 禁止一开始把重点放在图谱可视化或 agent 集成上。
- 先交付可运行 MVP，再做增强项。

### Hermes
- 负责维护需求边界、验收标准和里程碑口径。
- 审查功能是否偏离“收集 -> 整理 -> 召回”主闭环。
- 监督上层能力不要早于底层能力落地。

## 17. 当前拍板结论

本项目首版路线已经明确：
先做底层知识库能力，再逐步叠加知识图谱与 AI 上层建筑。

这意味着 v0.1 的优先级固定为：
1. 数据模型
2. 本地存储
3. 结构化编辑
4. 搜索与筛选
5. 相关条目推荐
6. 再考虑智能增强
